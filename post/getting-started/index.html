Version control systems.

General concepts

Version Control System (VCS) is used when several people work on one project. Usually, the main project tree is stored locally, in other words, the Version Control System (Version Control System, VCS) is software that allows you to track changes in documents, if necessary, roll them back, determine who and when made corrections, etc. The article discusses the types of VCS, the principles of their operation, and also provides examples of software products.

Centralized and distributed version control systems

  1. Centralized version control systems: These are client-server applications where a project repository exists in a single instance and is stored on the server. It was accessed through a special client application. Examples of such software products include CVS, Subversion.
  2. Distributed Version Control System (DVCS): It allows you to store a repository (a copy of it) with each developer working with this system. In this case, you can allocate a central repository (conditionally), to which changes from local ones will be sent and, with it, these local repositories will be synchronized. When working with such a system, users periodically synchronize their local repositories with the central one and work directly with their local copy. After you make enough changes to the local copy, they (the changes) are sent to the server. In this case, the server, most often, is chosen conditionally, because in most DVCS there is no such thing as a “dedicated server with a central repository”. The two most well-known DVCS are Git and Mercurial.

What are Git and GitHub?

Git is a version control system that has appealed to almost everyone, from developers to designers. GitHub can be considered a social network for storing code. This is a real Mecca for techies. Here you can practice development and come up with something of your own, find a lot of open-source projects, advanced technologies, various functions and designs. On GitHub, you learn and participate in other projects, store code for work or school, take the code of other projects and delve into all the details.

Networking and Server Solutions

Git uses the network only for exchange operations with remote repositories.

The following protocols can be used:

git-protocol (URI scheme - git:) - an open protocol that requires the presence of a running git daemon on the server (supplied with Git), the protocol does not have means of user authentication; SSH (ssh:) - uses user authentication using key pairs, as well as a built-in Unix-system “main” SSH server (sshd), on the server side it is required to create accounts with git as a shell; HTTP and HTTPS (http:, https:) ‚Äì uses the curl tool (for Windows , shipped with git), and its HTTP authentication capabilities, as well as its support for SSL and certificates. In the latter case, you need to work on the server side of a web application that acts as a layer between the Git commands on the server and the HTTP server (among them Is WebGitNet, developed on the MVC 4 ASP.NET). In addition to supporting the server-side push and pull commands, such web applications can also grant read-only access to the repository through a web browser.
